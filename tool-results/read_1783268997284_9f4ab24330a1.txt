   102→# 2 — SHARED SETTINGS ARCHITECTURE (applies to every tab)
   103→
   104→## 2.1 The one fetch/update pattern
   105→
   106→```
   107→READ    GET  /api/settings/:section?companyId=&year=   → SectionPayload (shape per tab below)
   108→WRITE   PUT  /api/settings/:section                    → { section, changes:[{key, value}], companyId?, reason? }
   109→        Response 200 { payload }   |  422 { fields:{key: message} }  |  403 | 409
   110→```
   111→
   112→Rules every write obeys:
   113→1. **Validate server-side** with the same validators the engine uses (`validatePAYERef`, `validateSortCode`, `modulusCheck`, `parseTaxCode`…). Client validation is UX only.
   114→2. **Audit every accepted change**: one `audit_log` row per key — `action:'SETTINGS_CHANGED'`, `entity_type:'settings:{section}'`, `before_json/after_json`, optional `reason`. Hash-chained as per PRD §11.2.
   115→3. **Optimistic UI with rollback**: apply locally → PUT → on 422 revert the field, show inline error; on network error revert + toast "Change not saved — retry".
   116→4. **Concurrency**: every SectionPayload carries `version` (max `updated_at` of underlying rows). PUT sends `If-Match: version`; mismatch → 409 → client refetches, shows "Settings changed by {actor} — review and retry".
   117→5. **Cache**: server components fetch fresh (no cache) — settings reads are cheap and correctness matters; after PUT, `router.refresh()`.
   118→
   119→## 2.2 Effective-dated configuration (the load-bearing rule)
   120→
   121→Statutory numbers are **never** stored as a single mutable value. They live in `statutory_config (tax_year, effective_from, key, value_json, source)` — PRD Part 4. Reading a rate = *latest row where `effective_from ≤ date`* for that year:
   122→
   123→```sql
   124→SELECT DISTINCT ON (key) key, value_json
   125→FROM statutory_config
   126→WHERE tax_year = $1 AND effective_from <= $2
   127→ORDER BY key, effective_from DESC;
   128→```
   129→
   130→```ts
   131→// server/config.ts
   132→export async function getStatutoryConfig(taxYear: string, onDate: string): Promise<Record<string, unknown>>
   133→export async function overrideStatutory(taxYear: string, key: string, value: unknown,
   134→  effectiveFrom: string, reason: string, actorId: string): Promise<void>
   135→// overrideStatutory INSERTS a new row (never UPDATE), audits CONFIG_OVERRIDE with reason (mandatory),
   136→// and invalidates the Redis key `cfg:{taxYear}` so the engine picks it up next calculation.
   137→```
   138→
   139→Consequence: recalculating March's pay run in June still uses March's numbers. This is the acceptance test in PRD §12.6.
   140→
   141→## 2.3 Bureau-level vs company-level scope
   142→
   143→Some settings are bureau-wide, some per-company. Every tab payload declares its scope; company-scoped tabs render a company selector (defaults to "Bureau defaults"):
   144→
   145→| Tab | Scope |
   146→|---|---|
   147→| Company | bureau defaults + per-company overrides |
   148→| Tax | global statutory (read) + per-company PAYE identity |
   149→| Pension | per-company scheme (+ bureau default template) |
   150→| Bank | per-company |
   151→| Users | bureau |
   152→| Security | bureau (+ per-company HMRC credentials) |
   153→| Compliance | bureau policy + per-company year-end status |
   154→| Notifications | bureau rules + per-user channel prefs |
   155→| System | bureau |
   156→
   157→Resolution order everywhere: `company override → bureau default → statutory/config constant`.

   163→## 3.1 COMPANY — `/bureau/settings/company`
   164→
   165→**Purpose:** bureau-wide defaults applied at client onboarding (your screenshot's "Company Defaults / Bureau-wide defaults applied to new client onboardings" card) + per-company overrides.
   166→
   167→**Payload:**
   168→```ts
   169→GET /api/settings/company?companyId=optional →
   170→{
   171→  scope: 'bureau' | 'company', version: string,
   172→  defaults: {
   173→    region: 'england_wales'|'scotland'|'northern_ireland',
   174→    paySchedule: { rule:'monthly_last_working_day'|'fixed_date'|'weekly'|'bi_weekly', day?: number },
   175→    earlyPay: boolean,
   176→    overtimeMultiplier: number,          // 1.5 default
   177→    pilonDivisor: 260|252|365,
   178→    holidayEntitlementBasis: 'statutory_5_6_weeks',
   179→    payslipTemplate: 'void_standard',
   180→    payrollIdPrefix: string,             // 'EMP-'
   181→  },
   182→  companies?: [{ id, name, overridesCount }]   // bureau view: which clients deviate
   183→}
   184→```
   185→
   186→**Logic:**
   187→- `Edit Defaults` opens an editor over `defaults`; PUT `{section:'company', changes:[...]}` → writes bureau row.
   188→- Changing a **default never retro-changes existing companies** — it seeds future onboardings only. Banner states this explicitly.
   189→- Per-company view (selector) shows the same fields resolved with override badges (`pearl dot = overridden`); clearing an override reverts to bureau default (`changes:[{key, value:null}]`).
   190→- Validation: `overtimeMultiplier ∈ [1, 3]`; `day ∈ [1,31]` only when rule=fixed_date; region change on an existing company warns "affects bank-holiday calendar and pay-date resolution from the next pay run".

   192→## 3.2 TAX — `/bureau/settings/tax`  (the tab in your screenshots' header strip)
   193→
   194→**Purpose:** view statutory thresholds per tax year, HMRC sync, controlled overrides, per-company PAYE identity.
   195→
   196→**Payload:**
   197→```ts
   198→GET /api/settings/tax?year=2026-27&companyId=optional →
   199→{
   200→  version: string,
   201→  years: { active:'2026-27', available:['2026-27','2025-26'] },
   202→  thresholds: [                                  // rendered as your Threshold table
   203→    { key:'personal_allowance',    label:'Personal Allowance',        value:12570,  unit:'£/yr',
   204→      priorValue:12570, variancePct:0, authority:'HMRC', effectiveFrom:'2026-04-06', overridden:false },
   205→    { key:'basic_rate_limit',      label:'Basic Rate Limit',          value:50270, ... },
   206→    { key:'higher_rate_limit',     label:'Higher Rate Limit',         value:125140, ... },
   207→    { key:'ni_pt',                 label:'NI Primary Threshold',      value:12570, ... },
   208→    { key:'ni_st',                 label:'NI Secondary Threshold',    value:5000, ... },
   209→    { key:'ni_uel',                label:'NI Upper Earnings Limit',   value:50270, ... },
   210→    { key:'ni_lel',                label:'NI Lower Earnings Limit',   value:6708, ... },
   211→    { key:'sl_plan_1'…'sl_plan_5','sl_postgrad', 'ae_trigger','ae_qel','ae_qeu',
   212→      'nlw_21','nmw_18_20','nmw_16_17','ssp_week','smp_std_week', ... }
   213→  ],
   214→  scotlandBands: [...], rukBands: [...],
   215→  sync: { lastRunAt:'2026-07-01T03:00:00Z', status:'ok'|'stale'|'failed', source:'gov.uk rates page' },
   216→  companyPaye?: { payeRef, aoRef, senderVerified: boolean }   // when companyId given
   217→}
   218→```
   219→
   220→**Logic — the three actions on this tab:**
   221→
   222→**(a) Update from HMRC (sync).** `POST /api/settings/tax/sync {year}` → 202 `{jobId}` → job `tax:sync`:
   223→```
   224→fetch canonical rates (v1: bundled JSON per year shipped with releases, "source: Kedbyte rates pack";
   225→                       v2: scrape/parse gov.uk rates-and-thresholds page)
   226→diff against current effective values
   227→  no changes  → sync.lastRunAt updated, toast "Thresholds verified — no changes"
   228→  changes     → DO NOT auto-apply. Create a review diff:
   229→                GET /api/settings/tax/sync/:jobId → [{key, current, incoming, source}]
   230→                Modal table with per-row Accept — accepted rows call overrideStatutory(...,
   231→                reason:'HMRC sync {date}'). Statutory numbers changing silently is how
   232→                payroll products create liabilities; a human always confirms.
   233→audit: TAX_SYNC_RUN + one CONFIG_OVERRIDE per accepted row
   234→```
   235→
   236→**(b) Override threshold.** Row action → modal: new value · `effectiveFrom` (default today; may be ≥ year start) · mandatory `reason`. `POST /api/settings/tax/override {taxYear, key, value, effectiveFrom, reason}` → inserts effective-dated row (§2.2), row shows `overridden:true` pearl badge + hover history (`GET /api/settings/tax/history?key=`). **Guard:** if committed pay runs exist with `pay_date ≥ effectiveFrom`, block with 409: "3 committed pay runs fall after this date. Overrides cannot rewrite committed history — corrections go through pay-run amendment (FPS correction)."
   237→- `Variance` column = `(value − priorYearValue)/priorYearValue × 100`, computed server-side, mono, amber if ≠ 0.
   238→
   239→**(c) Company PAYE identity** (company selected): `payeRef` (`^\d{3}/[A-Z0-9]{1,10}$`), `aoRef` (`^\d{3}P[A-Z]\d{7}[0-9X]$`). Editing either warns: "Changes apply to the NEXT RTI submission. Mid-year reference changes usually require an HMRC scheme transfer — confirm with HMRC first." Audited `PAYE_REF_CHANGED`.
   240→
   241→**Year selector:** switching to `2025-26` re-fetches read-only historical values (no override/sync buttons; banner "Closed tax year — read only").

   245→**Purpose:** per-company pension scheme configuration; bureau default template.
   246→
   247→**Payload:**
   248→```ts
   249→GET /api/settings/pension?companyId= →
   250→{
   251→  version: string,
   252→  scheme: { id, provider:'NEST'|'Peoples'|'Smart'|'Aviva'|'Other', schemeRef:string|null,
   253→            basis:'qualifying_earnings'|'pensionable_full'|'total_earnings',

Deadline chips: amber ≤ 14 days, red overdue. These same items feed the S02 dashboard Compliance card — one source function `getComplianceStatus(companyId, taxYear)`.
